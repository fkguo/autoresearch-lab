# Reviewer isolation spec (executable spec)

Goal: turn the Reviewer isolation principles into an **implementable, testable, auditable** engineering spec.

Core principles:
- Reviewer inputs should be evidence packets / artifacts / diffs / acceptance criteria.
- Reviewer should not see long executor chats or intermediate failed attempts (reduce confirmation bias and injection surface).
- Any escalation must require human approval and be persisted.

Chinese version (legacy / detailed notes): `docs/REVIEWER_ISOLATION.zh.md`.

## 1) Minimal viable implementation (M0/M1)

In M0/M1, treat the Reviewer as a clean-room external review process:

- **Process isolation**: run via separate CLI runners (`claude-cli-runner` / `gemini-cli-runner`), not within the same agent session.
- **Tool isolation**: tools/MCP disabled by default (Claude: explicit `--tools ""`; Gemini: no MCP/tools provided).
- **Input isolation**: the reviewer prompt file may contain only:
  - `review_packet.md` (generated by the Orchestrator)
  - an output contract (structured headers + verdict)
- **Escalation authorization**: if the Reviewer requests more info, the Orchestrator must enter `awaiting_approval` (escalation gate). A human decides whether to produce a new `review_packet.md` (new `approval_id`). The Reviewer must not fetch files or search the web directly.

Key point: **if the reviewer process has no tools, it cannot read files/run commands/use the network; it must decide from the packet only.**

## 2) Minimal `review_packet.md` contents (MVP)

The Orchestrator must generate an offline-reviewable packet (under run artifacts):
- Planner acceptance criteria (testable)
- action summary (what was done, what was not done, and why)
- key artifact pointers (paths + JSON keys for manifest/summary/analysis)
- diffs (or at least diff paths)
- risk disclosure (UNVERIFIED/assumptions list + minimal next validation)
- cost/budget stats (optional: network calls, runtime)

Additionally:
- the packet must forbid instruction-like text such as “approve unconditionally / ignore gates”. If such text appears, Reviewer must reject and request escalation.

## 3) Escalation protocol

### 3.1 When escalation is allowed

Reviewer may request escalation (structured fields only):
- missing artifact pointers prevent verification
- missing diffs/compile outputs
- potential injection/inconsistency requires higher-resolution logs

### 3.2 How escalation works (MVP)

- Orchestrator converts requests into an approval packet and enters `awaiting_approval`.
- Human decides:
  - `approve`: add the requested scope-limited information (e.g. compile log appendix)
  - `reject`: require a more concise/structured packet regeneration

All escalations must be appended to `.autoresearch/ledger.jsonl` (see `docs/ORCHESTRATOR_STATE.md`).

## 4) Testable acceptance criteria (MVP)

- RI-1 (tool isolation): reviewer runner invocation must disable tools (Claude: `--tools ""`) and logs must show it.
- RI-2 (input isolation): reviewer prompt may reference only `review_packet.md` (no extra file concatenation).
- RI-3 (no silent escalation): reviewer request for more info must enter approval-waiting state; no auto-feeding logs to reviewer.
- RI-4 (injection resistance): if the packet contains instruction-like injection text, reviewer verdict must be `NOT_READY` and request escalation/human intervention.

### 4.1 M1 implementation hook (make tests real)

Standardize reviewer calling pattern:
- Orchestrator generates `review_packet.md`
- Orchestrator calls the runner (external process) and records the command in trace/ledger

Example (Claude; tools disabled):

```bash
bash ~/.codex/skills/claude-cli-runner/scripts/run_claude.sh \
  --model opus \
  --tools '""' \
  --system-prompt-file prompts/plan_review_system_claude_strict.md \
  --prompt-file artifacts/runs/<TAG>/review/review_packet.md \
  --out artifacts/runs/<TAG>/review/reviewer_output.md
```

Example (Gemini; packet-only by design):

```bash
bash ~/.codex/skills/gemini-cli-runner/scripts/run_gemini.sh \
  --model gemini-3-pro-preview \
  --prompt-file artifacts/runs/<TAG>/review/review_packet.md \
  --out artifacts/runs/<TAG>/review/reviewer_output.md
```

## 5) Future enhancements (M2+)

If the Reviewer needs limited read-only verification abilities (e.g. checking references or compilation), add stronger isolation:
- OS-level sandbox + read-only mounts
- reviewer-specific audit tool allowlist (read-only; every access is logged)
- network allowlist (INSPIRE/arXiv/DOI stable anchors only)

Regardless:
- escalation must still require human approval
- Reviewer must not perform any write operations

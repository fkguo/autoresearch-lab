exhausted`：预算熔断触发\n- `running → early_stopped`：search policy 早停触发（与单次 `SearchStepResult.early_stopped` 区分：单次 step 的 early_stop_reason 可能只是局部熔断，不一定改变 campaign status）\n\n读写权限（强约束）：\n\n- **Read-only RPC**（`campaign.status` / `node.get` / `node.list`）必须在任何 status 下可用（除 `campaign_not_found`/`node_not_found`/`node_not_in_campaign` 外不得拒绝）。\n- **Side-effecting RPC**（`search.step` / `eval.run` / `rank.compute` / `node.promote`）仅在 `running` 允许；在 `paused|early_stopped|completed` 必须返回 `campaign_not_active`；在 `exhausted` 应优先返回 `budget_exhausted`（更具体）。\n\n关于 idempotency replay 与状态冲突（明确行为）：\n- duplicate hit 的 replay 必须回放首次响应（除 `idempotency.is_replay` 标志外一致），即使 campaign 当前 status 已变化；调用方不得用同一 `idempotency_key` 表达“新的意图”。\n\n**Post-early-stop / post-exhaustion 使用模式（informative）**：当 `search.step` 返回 `early_stopped=true`（无论是 step-local budget fuse 还是全局预算耗尽/策略早停），调用方 **不得**用同一个 `idempotency_key` 重试期待“继续做剩下的工作”，因为这会 replay 原部分结果。若要在 `campaign.topup` 或策略调整后继续探索，调用方必须以新的 `idempotency_key` 重新调用 `search.step`。\n\n## 3. 核心抽象：Operator + SearchPolicy + Distributor（可替换）\n\n> 把“科学发现路径”落到可执行机制：Operator 是“如何变异/扩展 idea”；SearchPolicy 是“如何在树/种群中调度探索”；Distributor 是“预算如何分配到不同 LLM 与不同 operator/island”。\n\n> 增补（2026-02-12）：为实现“多 agents 物理学家社区/组团探索”，我们引入 **Team/Role 作为可插拔运行时拓扑**：同一个 island/方向不再对应单一生成器，而是对应一个由角色组成的团队（并行/串行可配置），其输出通过 artifacts 汇聚回 IdeaStore。\n\n### 3.1 `Operator`（发现算子）\n\n**职责**：输入一个 seed/idea-node，输出若干候选（带 rationale 草稿），并附带操作痕迹（trace）。\n\n建议最小接口（概念）：\n- `operator_id` / `operator_family`\n- `apply(input, context) -> [RationaleDraft]`\n- `operator_trace`：包含策略参数、随机种子、引用的证据 URI（用于审计/复现/回放）\n\n**Operator families（v0.2 建议）**  \n（把科学史/科学哲学转成可执行算子族；HEP pack 可先实现其中 4–6 个）\n\n- `AnomalyAbduction`：反常 → 解释（Kuhn/Peirce abduction）\n- `AssumptionInversion`：假设反转（Popper；增强可证伪性）\n- `SymmetryOperator`：对称性操作（破缺/恢复/推广/对偶）\n- `LimitExplorer`：极限外推（强/弱耦合、能标极限、维数极限）\n- `CrossDomainAnalogy`：结构/方法/现象映射（强制输出 mapping table；跨学科/跨分支方法迁移的一等公民）\n- `CombinatorialSynthesis`：方法×理论模块组合（IdeaSearch 风格再组合）\n- `ProtectiveBeltPatch`：Lakatos（保持 hard core，改 protective belt）\n- `RepresentationShift`：改变表述（变量替换、对偶变换、规范选择）\n\n### 3.2 `SearchPolicy`（搜索策略）\n\n**职责**：在预算约束下，决定“扩展哪个节点/哪个岛、使用哪个 operator、保留哪些候选”。\n\nv0.x 推荐先支持：\n\n1. **Divergent–Convergent 外环**（管线纪律）\n2. **Multi-Island Evolution 内核**（IdeaSearch 风格：不同策略群体并行 + repopulate）\n3. 可选：**BFTS / BeamSearch**（对树结构分支的局部 best-first）\n\n> 设计要点：SearchPolicy 只依赖 `IdeaNode` 与 `Evaluator` 的输出，不依赖领域细节。\n\n#### 3.2.1 Multi-Island 状态机（最小可实现规格）\n\n为避免“同名多岛”但实现不可互换，v0.2 要求至少声明以下状态与迁移：\n\n- **States**：`SEEDING → EXPLORING → CONVERGING → STAGNANT → REPOPULATED | EXHAUSTED`\n- **Triggers（示例）**：\n  - `SEEDING → EXPLORING`：population 达到 `min_pop_size`\n  - `EXPLORING → CONVERGING`：top-k 语义多样性低于阈值（避免模式坍缩）\n  - `EXPLORING/CONVERGING → STAGNANT`：`best_score_improvement(last_n_steps) < ε`\n  - `STAGNANT → REPOPULATED`：执行 repopulate（从 donor islands 注入 migrants）\n  - `STAGNANT → EXHAUSTED`：预算不足以完成一次最小 step（`budget_remaining < min_step_cost`）\n  - `REPOPULATED → EXPLORING`：重置 stagnation 计数器后继续\n\n需要对外暴露可替换的谓词/策略接口（概念）：\n- `should_repopulate(island_state, history, budget_remaining) -> bool`\n- `migrate(donor_islands, target_island, n_migrants, selection_fn) -> IdeaNode[]`\n\n### 3.3 `Distributor`（预算分配/调度器）\n\n**职责**：把生成请求分配给：\n\n- 不同 LLM backends（异构模型表型互补：explorer vs converger vs critic）\n- 不同 islands / operator families\n\n推荐 v0.2 就引入 **softmax-bandit**（IdeaSearchFitter 给出可复用机制）：\n\n- 为每个 backend/operator 维护近期 reward 的 EMA\n- 用 `softmax(score/T)` 分配生成配额（避免单点收敛）；更具体地，可采用：\n  - `p_i = exp(s_i / T_model) / Σ_j exp(s_j / T_model)`  \n  - 其中 `s_i` 为后端/算子的 reward EMA，`T_model` 控制探索-利用权衡\n\n**Budget Circuit Breaker（全局熔断器，必须）**：\n- 当 `total_cost > max_budget` 或 `total_tokens > max_tokens` 或 `wall_clock_s > max_wall_clock_s` 时，立即终止所有 pending 生成/评估，进入 `EarlyStop`，并写入 `budget_checkpoint` ledger 事件。\n- 建议提供 `degradation_order[]`：当预算紧张时的降级顺序（例如先减少评估轮数，再减少 islands，再禁用跨域算子）。\n\nreward 信号分两类：\n- **短期 proxy**：grounding ratio、novelty proxy、结构化可编译率（是否能形成 Canonical IdeaCard）\n- **长期 outcome**：A0.2 通过率、C2 成功率、W_compute 成功率（延迟奖励）\n\n**Phenotype profiling（用于异构分工与调度，不是“模型排行榜”）**：\n- 迁移自 IdeaSearchFitter 的动态过程指标：`Iter@k`（不同 epoch/step 的累积成功）、epoch 分布（中位数/3-4 分位）、成本（word/token）。\n- ideation 侧建议最小映射：`A0_pass@k`、`epochs_to_first_grounded_claim`、`tokens_per_promoted_node`、`checker_disagreement_rate`。这些指标应写入 `origin`/`eval_info` 以便回放与调度学习。\n\n### 3.4 Physicist Community：Team/Role 组团探索（OpenClaw Broadcast Groups 启发）\n\n> 目的：把“多 agent 协作”从泛泛口号变成可配置、可审计、可扩展的运行时拓扑；同时避免把评审/审查逻辑硬编码进某个单体提示词。\n\n#### 3.4.1 核心概念（最小可实现）\n\n- **Role**：一个“具名职责 + 工具/模型权限 + 输出契约”的执行单元（可由不同 LLM 或工具沙箱承载）。\n- **Team**：一组 Roles，围绕同一研究方向/策略 island 协作；Team 有协调策略（并行/串行/分阶段）。\n- **Community**：多个 Teams 的集合 + 中央仲裁/汇总角色（用于跨团队排名、repopulate、冲突裁定）。\n\n建议把 multi-island 解释为：`island_id` 同时标识\n1) 搜索策略群体（operator/constraints 权重配置）与\n2) 该策略群体对应的 Team（role composition + coordination policy）。\n\n#### 3.4.2 角色建议（HEP-first 的最小社区）\n\n> v0.2 不要求一次性全实现；但需要把“角色接口”设计出来，后续可逐步替换为更强的执行体。\n\n- `Ideator`（发展想法）：发散生成、提出机制与测试点（输出 `RationaleDraft`）。\n- `Librarian`（证据检索）：INSPIRE/PDG/讲义/综述检索，产出 `evidence_packet`（URI + 摘要 + 相关性）。\n- `Formalizer`（结构化形式化）：把 rationale 收敛为 `IdeaCard`（schema-validated）。\n- `Derivation`（推导/一致性检查）：对称性/量纲/极限/一致性快速检查，产出 `consistency_report`（可带最小手算/公式）。\n- `Coder`（最小计算原型）：把 compute plan 落到可执行草图（最小脚本/伪代码/数值检查路径）。\n- `Checker`（复核/复现）：独立重算/重查（clean-room），给出一致/不一致结论与最小反例。\n- `Referee`（可行性/新颖性/影响审查）：按 rubric 打分，但必须输出“**创新增量**”而非修辞差异（见 6.2）。\n- `Editor`（汇总/归档）：把各角色结构化输出合并为 `IdeaNode` 更新（trace 完整、可回放）。\n\n> Packaging 建议：`Referee` 不建议作为“与 idea-generator 并列的独立产品 agent”。更合理的做法是：把它当作 **idea-generator 的可插拔 role**（可由独立 session/独立模型承载，以获得 clean-room 效果与更严格的工具权限），从而同时满足“模块化替换”和“端到端门禁一致性”。\n\n#### 3.4.3 协调策略（并行 vs 串行 vs 分阶段）\n\n三种基本模式（可作为 `TeamPolicy` 的枚举）：\n\n1. **Parallel Divergence**：并行广播给多个 roles（尤其是多个 ideator/referee），提高多样性与盲点覆盖。\n2. **Sequential Convergence**：`Librarian → Ideator → Formalizer → Referee`，用证据与结构化输出抑制幻觉与格式漂移。\n3. **Stage-Gated**：在关键门禁前强制进入特定阶段（如 promote 前必须跑 `Checker` 与 `GroundingAudit`）。\n\n> 关键纪律：默认 **Role clean-room**；role 之间不共享草稿，只共享输入上下文包与最终 artifacts。若触发辩论，则以 `debate_packet`（point/counterpoint + evidence_uris）形式显式发生。\n\n> **成本感知约束**：Team topology 的 token 消耗近似为 `Σ(roles) × per-role-cost`；Distributor 在分配预算时必须将 team composition 纳入每步成本估算（而非假设每步 = 1 次 LLM 调用）。`BudgetEnvelope.extensions` 可承载 `team_cost_multiplier`/`role_cost_table` 等运行时参数。\n\n---\n\n## 4. 强制两阶段：RationaleDraft → Canonical IdeaCard（Explain-Then-Formalize）\n\n### 4.1 为什么必须强制\n\nExplain-Then-Formalize 的价值在于：\n\n- 允许 **发散**（类比、隐喻、反转假设）  \n- 但必须在进入门禁/下游前完成 **形式化**（可验证字段齐全）\n\n### 4.2 两阶段产物（建议）\n\n1. `RationaleDraft`（允许高温）：\n   - WHY：动机/反常/类比映射表/机制猜想\n   - 风险：潜在已知/folklore、物理一致性风险、验证优先级\n   - 下一步最小验证：1–3 个 kill criteria\n\n2. `IdeaCard`（低温约束 + schema 校验）：\n   - `thesis_statement`\n   - `testable_hypotheses[]`\n   - `required_observables[]`\n   - `candidate_formalisms[]`（必须映射到 DomainPack 的 formalism registry；否则不得 promote 到 C2）\n   - `minimal_compute_plan[]`\n   - `claims[]`（claim-level 溯源；每条 claim 至少包含 support_type + evidence_uris；对 `llm_inference/assumption` 必须给 verification_plan；可选 verification_status）\n\n> 硬规则：任何 idea 进入 Ranking / A0.2 前，必须完成 `IdeaCard` 生成与 schema 验证。\n\n#### 4.2.1 Grounding Audit Gate（IdeaCard 生效前的强制门禁）\n\n为避免“看似有引用但引用并不支撑 claim”的失真 provenance，IdeaCard 只有在通过 grounding audit 后才被视为“有效可推广对象”：\n\n1. **URI 可解析（active resolution）**：`claims[].evidence_uris[]` 必须通过 **active lookup** 验证可解析（INSPIRE API / DOI resolver / KB artifact existence check 等），不得仅做格式校验；不存在 phantom 引用。解析失败必须写入 `IdeaNode.grounding_audit.failures[]`，并将 `status` 置为 `fail` 或 `partial`。\n2. **数据一致性**：`support_type=data` 的数值类 claim，必须与 PDG/HEPData 在约定容差内一致（否则标记 FAIL）。\n3. **推断透明**：`support_type=llm_inference/assumption` 必须有 `verification_plan`（至少 1 条 kill criterion）。\n4. **folklore 预筛**：产出 `folklore_risk_score ∈ [0,1]`；超过阈值则必须走 `A0-folklore` 人类裁定。\n5. **晋升门禁（强约束）**：`node.promote` 的成功条件必须包含 `grounding_audit.status == pass`。`partial/fail` 一律阻塞晋升（返回 `grounding_audit_failed`），避免“带着缺口进入 C2”。\n\n审计输出写入 `IdeaNode.grounding_audit`（并可汇总进 `idea_evidence_graph_v1.json`）。Grounding Audit 的默认触发点应当是 `eval.run`（当 `EvaluatorConfig.dimensions` 包含 `grounding` 时）；也允许由 `Checker` 角色在生成后即时执行（以加速剪枝）。\n\n---\n\n## 5. agent）\n\n维度建议延续：\n- novelty / feasibility / impact / tractability / grounding\n\n关键工程约束：\n- evaluator 返回不只是分数，还要返回 **可再投喂的诊断**（参考 IdeaSearch-framework 的 score+info）\n- 多 agent 评审默认 **clean-room**（互不共享对话记忆），直到触发“结构化辩论”才允许受控信息流\n\n#### 6.1.1 clean-room 与结构化辩论（最小协议）\n\n- clean-room：每个 evaluator 在独立会话/上下文中完成评分与证据检索（不共享草稿、不中途互相影响）。\n- 触发辩论：任一维度分歧超过阈值（例如 `|Δscore| > 2`）或出现互斥结论（“已发表/未发表”）。\n- 辩论输出必须结构化：point/counterpoint + evidence_uris + 最终裁定理由（作为 `eval_info` 的一部分写回）。\n\n### 6.2 Novelty：四层栈 + folklore 风险\n\n沿用 2026-02-11 的四层 novelty pipeline，但把 folklore 风险变成显式字段：\n- `folklore_risk`（高则必须走 `A0-folklore` 人类裁定）\n\n**重要补充：避免把“细枝末节”当成创新（Referee 硬约束）**\n\n当 `EvaluatorConfig.dimensions` 包含 `novelty` 时，负责 novelty 维度的 Referee 输出必须包含一个 `novelty_delta_table`（可作为 `eval_info` 的结构化字段或独立 artifact），至少回答：\n\n1. **closest prior**：最相近的 1–3 篇/方向（URI + 一句摘要）\n2. **delta type**（枚举，可多选）：`new_mechanism | new_observable | new_regime | new_method | new_formalism | new_dataset | new_constraint`\n3. **delta statement**：每个 delta 必须是一句“可被证伪”的陈述（不是措辞变化）\n4. **non-novelty flags**：显式标注以下情况不计为主要创新：参数微调、符号替换但等价、仅换叙述不换预测、仅合并已知组件但无新可检验结论\n5. **verification hook**：每个 delta 关联一个最小验证（可文献/可计算/可观测）\n\n### 6.3 Ranking：Pareto + Tournament（Elo）\n\n- Pareto：多目标前沿保证不被单一分数绑架\n- Tournament/Elo：降低评分尺度漂移影响，促进相对比较的稳定性\n  - **硬约束**：Elo 必须是 **bounded + deterministic** 的（否则成本不可控/不可复现）。建议 `rank.compute(method=elo)` 接受 `elo_config={max_rounds, seed}`，并把 pairing 规则写入 ranking artifact 以便回放。\n\n---\n\n## 7. DomainPack（领域插件）最小化设计（避免过度抽象）\n\nv0.2 建议把扩展点收敛为 6 类（HEP 先硬编码实现，再抽取接口）：\n\n1. `seed_sources`\n2. `operators`（或 operator 参数化/模板）\n3. `constraints_and_validators`\n4. `retrieval_recipes`（INSPIRE 查询模板、关键词扩展、分类映射）\n5. `feasibility_estimators`\n6. `method_compilers`（IdeaCard → C2 handoff；必须可校验）\n\n> DomainPack 的“知识载体”主要是：ontology + prompt templates + validators，而不是把物理写成大量 Python 规则（可维护性更好）。\n\n**重要：HEP-first ≠ HEP-only（跨学科方法迁移必须预留接口）**  \n许多突破来自“方法/表述/直觉”的跨域迁移，而不是同一子领域内的微调。因此在架构层必须保证：\n- `domain` 只是 campaign 的 **primary target domain**（用于验收/约束/评估权重），不应把检索与算子空间硬锁死在 HEP。\n- 支持“可插拔的 MethodPacks / CrossDomainPacks”：它们提供跨学科的 `operators + retrieval_recipes + validators`（例如统计物理/信息论/数学工具/ML），由 `TeamPolicy`/预算降级策略显式启用/禁用（见 `disable_cross_domain_operators`）。\n- 跨域算子必须产出显式 `mapping table + invariants + kill criteria`（避免隐喻式类比污染），并通过 grounding audit（引用可解析）与目标域硬约束（量纲/对称性/可证伪性）。\n\n**formalism registry（为 C2 提供可校验映射）**：\n- DomainPack 必须声明 `formalism_id → {c2_schema_ref, validator, compiler}` 的映射。\n- `candidate_formalisms[]` 必须来自该 registry；否则 `node.promote` 必须失败（schema_validation_failed 或 grounding_audit_failed 或 formalism_not_in_registry）。\n\n---\n\n## 8. 与现有生态圈集成（HEP-Autoresearch）\n\n### 8.1 输入（seed sources）\n\n- C1 gaps（系统性缺口）\n- KB priors（已有笔记/失败记录/方法痕迹）\n- PDG/HEPData tensions（反常/张力）\n- user seeds（`seeds.yaml` / `ideas.md`）\n\n### 8.2 输出（handoff）\n\n`idea_handoff_c2_v1.json` 是唯一允许进入 C2 的入口：\n- 缺字段 → 直接拒绝（不可“口头交接”）\n- **Schema SSOT**：`schemas/idea_handoff_c2_v1.schema.json`（由 `node.promote` 产出；C2 侧必须以此做机读校验）\n\n### 8.3 门禁（A0）\n\n沿用双层：\n- `A0.1` Campaign charter（方向/预算/风险）\n- `A0.2` Idea promotion（选具体 idea 投入资源）\n\n---\n\n## 9. 里程碑（v0.2 → v0.3）\n\nv0.2（本 spec）交付“架构与契约”优先：\n\n1. 固化 artifacts 与 IdeaNode 字段（含 origin/operator_trace/eval_info/grounding_audit）\n2. 交付机器校验 schema（`schemas/*.schema.json` + OpenRPC）\n3.

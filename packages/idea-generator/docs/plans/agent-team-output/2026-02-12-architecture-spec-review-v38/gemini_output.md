## Blockers
None. The architecture is highly mature, with rigorous idempotency, a well-defined state machine, and a clear "evidence-first" approach to scientific discovery.

## Non-blocking
1. **Reduction Audit Strictness**: In `reduction_audit_v1.schema.json`, a `skipped` toy check (allowed if Coder/Checker roles are unavailable) correctly results in a `partial` status. Since `node.promote` requires `status: pass`, this means promotion is effectively blocked until a toy check is successfully executed. This is a strong safety choice for v0.x but may require a "manual override" or "human-in-the-loop pass" for specific edge cases in future versions.
2. **Registry Uniqueness**: The OpenRPC specification for `campaign.init` requires the engine to reject duplicate `abstract_problem_type` values. While the JSON Schema for the registry doesn't (and shouldn't) enforce this globally, implementers must ensure the merged registry logic handles collisions deterministically (caller-first as specified).
3. **Elo Determinism**: `rank.compute(method=elo)` relies on `elo_config.seed`. Implementers should ensure the pairing algorithm is strictly deterministic given the seed and the sorted node list to ensure auditability.

## Real-research fit
1. **Abstraction & Reduction**: The `ProblemReduction` and `TechniqueTransplant` operators directly address a core research pattern: mapping a novel physics problem to established mathematical or computer science solutions (e.g., optimization, graph theory). This prevents "reinventing the wheel" and grounds ideas in mature formalisms.
2. **Physicist Community**: The Role/Team/Community hierarchy (Ideator, Librarian, Checker, etc.) mirrors the peer-review and collaboration cycles in high-energy physics.
3. **Novelty Delta Tables**: Requiring a `novelty_delta_table` to distinguish between mechanisms, observables, and regimes effectively mitigates the risk of "innovation-by-thesaurus" common in naive LLM brainstorms.

## Robustness & safety
1. **Idempotency**: The requirement for JCS (RFC 8785) canonicalization for payload hashing is excellent. It ensures that semantically identical requests (even with differing key order) are correctly deduplicated.
2. **Budget Circuit Breaking**: The pre-tick budget check in the Distributor (refusing to start if estimated cost > remaining) is a critical safeguard against half-finished, expensive "physicist team" computations.
3. **Provenance Chain**: The mandatory inclusion of `has_reduction_report` and `reduction_audit_summary` in the `promotion_result` allows for high-level machine auditing of the handoff without deep artifact inspection.

## Specific patch suggestions

1. **`schemas/reduction_audit_v1.schema.json`**: The current `partial` status condition is too restrictive (it requires *all* assumptions to be pending). Update the logic to allow a mix of `satisfied` and `pending_verification`.
   - **Patch**: In the `allOf` block defining `status: partial`, change the `assumptions` condition from `not: { contains: { ... status: "satisfied" } }` to simply ensure it `not: { contains: { ... status: "violated" } }` and then require that it `contains: { ... status: "pending_verification" }` OR that `toy_check_result` is `skipped`.

2. **`docs/plans/2026-02-12-idea-generator-architecture-spec.md`**: In Section 4.2.1 (Grounding Audit Gate), clarify that `status: partial` is not just a warning but a promotion-blocking state.
   - **Patch**: Append to point 5: "Nodes with `status: partial` (e.g., due to pending assumptions or skipped toy checks) are considered 'in-progress' and MUST be updated via a new revision with a passing audit before promotion can occur."

3. **`schemas/idea_core_rpc_v1.openrpc.json`**: Add an explicit cross-reference in the `rank.compute` description to `eval.run` to clarify the data flow for scorecards.

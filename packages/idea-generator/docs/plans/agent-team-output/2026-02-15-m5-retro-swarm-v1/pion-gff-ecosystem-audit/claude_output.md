**Summary: The ecosystem demonstrates strong control-plane discipline but the compute-plane is a toy random sampler mislabeled as "bootstrap," with numerics below research grade, literature gating that rewards quantity over relevance, no mechanism for idea generation, and a repo layout that splits artifacts across too many parallel trees.**

## Method Fidelity

- **Severity: H** | **Component: compute-plane** | **Location: `research/pion_gff_bootstrap/src/PionGFFBootstrap.jl` (entire `run` function)** | **Problem:** The method is Monte Carlo random sampling of polynomial coefficients with sigmoid compression — it enforces zero physics constraints (no crossing symmetry, no Roy equations, no Froissart bound, no Adler zero, no ChPT matching) and therefore cannot be called "bootstrap" in the sense of [R2]–[R4].** | **Suggestion:** Add a gate field `method_classification` to the results schema (`"random_sampling" | "constrained_bootstrap" | "dispersive"`) and require the M0.5 gate to reject `random_sampling` as the sole method when the project claims "modern bootstrap."

- **Severity: H** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, lines defining `d0 = d0_base + alpha * i0`** | **Problem:** The D-term `D^π(0)` is generated as `Uniform(d0_min, d0_max) + Uniform(alpha_min, alpha_max) * I_0`, which has no connection to the actual coupled unitarity relation `Im D^π` in [R1] Eq.(ImDpi); the output bounds are artifacts of the prior range, not physics.** | **Suggestion:** Require the results schema to carry a `constraints_enforced` list; the gate should check that at least `["elastic_unitarity_A", "elastic_unitarity_D"]` are present before accepting bounds as "bootstrap-derived."

- **Severity: H** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `_phase_integral` and phase construction** | **Problem:** The phase-shift ansatz `δ(t) = δ_max · x · sigmoid(poly(x))` is not constrained to satisfy `δ_0^0(t_thr) → a_0^0 · σ_π · (t−t_thr)^{1/2}` (threshold behavior) or any sum-rule / Roy-equation crossing constraint, so the "allowed family" is far wider than the physical bootstrap region.** | **Suggestion:** Implement at minimum a threshold-behavior filter (reject samples where `δ(t)` near threshold doesn't scale as `σ_π^{2ℓ+1}`) and add a `constraint_violations` count to results; gate rejects if violation rate > configurable threshold.

- **Severity: M** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `run` function, `p1 = rand_uniform(rng, p1_min, p1_max)`** | **Problem:** The subtraction constant `α` (mapped to `p1`) is sampled uniformly from a configurable prior rather than being determined or bounded by data/ChPT, making the output `r_g^2` range a reflection of the chosen prior width.** | **Suggestion:** Add a `prior_source` field to config and results (`"chpt_nlo" | "lattice" | "flat_scan"`) and require the gate to flag `flat_scan` as needing explicit prior-sensitivity analysis.

- **Severity: M** | **Component: workflow** | **Location: `docs/research/pion-gff-bootstrap/m0.2-design.md`, §4.3** | **Problem:** The design doc promises "generate an allowed family of elastic partial waves using a laptop-feasible bootstrap ansatz inspired by [R2]–[R4]" but the implementation samples random polynomials with no bootstrap constraint whatsoever — the design-to-implementation traceability is broken.** | **Suggestion:** Add a `design_implementation_delta` artifact (machine-generated diff between design promises and implemented constraints) as a required gate input for M0.3+.

## Literature & Evidence

S-matrix bootstrap, Córdoba–Guerrieri–Hebbar dual bootstrap bounds on form factors); the literature gate `new_retrieval_count >= 5` rewards quantity, not relevance.** | **Suggestion:** Add a `relevance_tag` enum (`"bootstrap_method" | "dispersive_method" | "model_computation" | "experimental"`) to each included paper and require at least N papers tagged `bootstrap_method` in the gate (`min_bootstrap_method_count >= 2`).

- **Severity: M** | **Component: hepar** | **Location: `artifacts/literature/search_evidence.json`, `queries`** | **Problem:** Search queries target "pion gravitational form factor" but never search for "S-matrix bootstrap pion" or "form factor bootstrap bounds" or "positivity bounds gravitational form factor," missing the actual bootstrap literature.** | **Suggestion:** Require the literature search spec to include at least one query per seed-paper methodology family (bootstrap, dispersive, lattice) and record which families were queried in a machine-checkable `query_coverage` field.

- **Severity: M** | **Component: hepar** | **Location: `artifacts/literature/seed_papers.json`** | **Problem:** Seed papers [R2]–[R4] are about ππ scattering amplitude bootstrap, not about gravitational form factor bootstrap specifically; no seed paper on positivity/bootstrap bounds for GFFs (e.g., Córdoba–Guerrieri–Hebbar 2308.xxxxx, or Bern et al. graviton exchange positivity) is included.** | **Suggestion:** Add a `seed_gap_analysis` artifact that lists which sub-topics of the project are not covered by any seed and flags them for manual expansion before M0.3 gate.

- **Severity: L** | **Component: hepar** | **Location: `artifacts/literature/search_evidence.json`, `filtered_out`** | **Problem:** Only one paper is listed as filtered out with a vague reason ("Kept as background context but not selected"), providing no evidence of systematic triage.** | **Suggestion:** Require `filtered_out` entries to carry a structured `exclusion_reason` enum (`"off_topic" | "superseded" | "duplicate" | "insufficient_novelty"`) and record total hit count vs. included count.

## Numerics

- **Severity: H** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `_phase_integral` function** | **Problem:** Uses composite trapezoidal rule on a uniform grid for an integrand `φ(t)/t²` that diverges as `t → t_thr` (since `t_thr ≈ 7.8×10⁴ MeV²` and `φ` starts from 0, the integrand is small at threshold but `1/t²` weighting makes mid-range accuracy poor); no error estimate is produced, and the M0.2 design doc §3.1 specifies a `quad` config field for integration tolerances that was never implemented.** | **Suggestion:** Replace with Gauss-Legendre or adaptive quadrature (e.g., `QuadGK.jl`), emit a `quadrature_error_estimate` field in results, and add a gate check that the error estimate is below a configurable tolerance.

- **Severity: H** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `run`, line `grid_n = max(grid_n, 64)`** | **Problem:** The default 512-point uniform grid with trapezoidal rule gives O(h²) error ≈ O(1/512²) ≈ 4×10⁻⁶ relative to the integration range, but no convergence study (grid doubling) is performed per-run; the M0.4 robustness campaigns vary `elastic_t_max` and `n_params` but never `grid_n`.** | **Suggestion:** Add `grid_n` as a required sweep dimension in M0.4 campaigns and emit `quadrature_convergence_ratio` (ratio of result at grid_n vs grid_n/2) in results; gate rejects if ratio deviates from 1.0 by more than a threshold.

- **Severity: M** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `sigmoid` function** | **Problem:** The sigmoid implementation has a branch for numerical stability (`x >= 0` vs `x < 0`), but `_poly_horner` can produce very large values (unbounded `coeff_scale * randn`), leading to sigmoid saturation at 0 or 1 for most samples — effectively the phase is either 0 or `δ_max · x`, reducing the "family" to a trivial binary.** | **Suggestion:** Add a diagnostic `sigmoid_saturation_fraction` (fraction of grid points where `|p(x)| > 10`) to results and flag in the gate if saturation exceeds 50%.

- **Severity: M** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `n_samples = 200`** | **Problem:** 200 random samples is insufficient to characterize the extremes of a distribution over a 6-dimensional coefficient space (2 × n_params coefficients + 3 uniform parameters); the min/max "bounds" are under-sampled envelope estimates with no confidence interval.** | **Suggestion:** Either increase default `n_samples` to ≥ 10⁴ or switch to an optimization-based extremization (e.g., `Optim.jl` to find actual min/max of the observables subject to constraints); report `n_samples` and a bootstrap confidence interval on the min/max.

- **Severity: L** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, `_poly_horner`** | **Problem:** Horner evaluation is correct but uses 1-based reverse iteration that could be confusing; more importantly, no overflow/underflow guard exists for high-degree polynomials with large coefficients.** | **Suggestion:** Add a `max_poly_value` diagnostic and clamp or reject samples where `|p(x)| > threshold` before computing sigmoid.

## Gates & Contracts

- **Severity: H** | **Component: hepar** | **Location: `docs/research/pion-gff-bootstrap/runs/2026-02-15-m0.5-final-v1/live-v1/pipeline.json`, `gate.payload`** | **Problem:** The M0.5 gate payload embeds the final numerical bounds directly (copy-pasted from results), but there is no gate check that these bounds are *physically reasonable* — e.g., `r_g² ∈ [0.007, 0.321]` fm² spans nearly two orders of magnitude, which would be a trivially uninformative bound for any real analysis, yet the gate passes.** | **Suggestion:** Add a `bound_informativeness` gate check: `(max-min)/|max+min| < configurable_threshold` (e.g., < 5.0), or at minimum require that the bound width is compared to a reference value from literature and recorded in the gate payload.

- **Severity: H** | **Component: hepar** | **Location: gate schema (referenced as `pion_gff_bootstrap_m04_gate_v1.schema.json` and `m05_gate_v1.schema.json` but not provided)** | **Problem:** Gate schemas are referenced but not included in the audit packet, making it impossible to verify what is actually enforced; the contract between design docs and implementation is unauditable.** | **Suggestion:** Always include gate schemas as committed artifacts in `contracts/` and add a CI check that every `schema_path` referenced in any `pipeline.json` exists in the repo.

- **Severity: H** | **Component: workflow** | **Location: all pipeline.json files, `gate.payload.novelty`** | **Problem:** The novelty gate points to `artifacts/ideas/m03-control-plane-pipeline/novelty_delta.json` for all milestones (M0.3, M0.4, M0.5), meaning novelty was assessed once at M0.3 and never re-evaluated as the method and results evolved — this is a stale-evidence problem.** | **Suggestion:** Require the novelty artifact path to be milestone-specific (`novelty_delta_m04.json`, `novelty_delta_m05.json`) and require re-evaluation at each milestone gate.

- **Severity: M** | **Component: hepar** | **Location: `m0.2-design.md` §3.1 vs `PionGFFBootstrap.jl`** | **Problem:** The config schema specifies `constraints` (crossing/analyticity toggles) and `quad` (integration tolerances) fields that are never read or enforced by the Julia code; the I/O contract is partially fictional.** | **Suggestion:** Add a CI test that parses the config schema and verifies every documented field is read by `_nested_get` in the Julia code (or explicitly documented as "reserved/unimplemented").

- **Severity: M** | **Component: hepar** | **Location: `PionGFFBootstrap.jl`, `run` return dict, `provenance`** | **Problem:** The `provenance` dict records `julia_version` and `active_project` but not the git commit SHA, contradicting the M0.2 design which requires "git commit (compute repo)" in `results.json`.** | **Suggestion:** Add `git_sha` to the provenance block (call `read(pipeline("git rev-parse HEAD"), String)` or pass it via config) and add a gate check that `provenance.git_sha` is non-null.

- **Severity: M** | **Component: workflow** | **Location: `m0.4-robustness.md`, §B "Required sweep families"** | **Problem:** The design requires 5 sweep families but only 3 produce physics bounds (truncation, param_sensitivity, seed_nparams); retry_injection and resume_proof are control-plane tests that should not be listed as robustness campaigns in the same gate payload — conflating infrastructure testing with physics robustness.** | **Suggestion:** Split the gate into `robustness.physics_campaigns` and `robustness.infrastructure_proofs` with separate minimum counts and validation logic.

- **Severity: L** | **Component: hepar** | **Location: all pipeline.json, `review.claude_system_prompt_file`** | **Problem:** The system prompt path is an absolute local path (`/Users/fkg/Nutstore Files/...`) which breaks reproducibility on any other machine.** | **Suggestion:** Vendor the system prompt into the repo (e.g., `contracts/review-system-prompt.md`) and reference it with a relative path; gate should reject absolute paths outside the workspace root.

- **Severity: L** | **Component: hepar** | **Location: pipeline.json, `review.convergence`** | **Problem:** M0.3 allows 100 review rounds with 5760 minutes (4 days) wall clock, while M0.4 allows 12 rounds / 360 min — the convergence parameters vary wildly across milestones with no documented rationale, suggesting they were tuned ad hoc.** | **Suggestion:** Define default convergence parameters in a shared config (e.g., `contracts/review-convergence-defaults.json`) and require per-milestone overrides to carry a `rationale` string.

## Repo UX

- **Severity: M** | **Component: repo-UX** | **Location: `docs/research/pion-gff-bootstrap/runs/` vs `artifacts/runs/pion-gff-bootstrap/`** | **Problem:** Pipeline run metadata lives under `docs/research/.../runs/` while campaign outputs live under `artifacts/runs/...`, splitting a single logical run across two trees with no cross-reference beyond prose in evidence-index files.** | **Suggestion:** Adopt a rule: all machine-generated outputs go under `artifacts/`; `docs/` contains only human-authored documents and links; add a repo-layout linter in CI.

- **Severity: M** | **Component: repo-UX** | **Location: `docs/research/pion-gff-bootstrap/pion-gff-bootstrap-paper.zh.md`** | **Problem:** The main human-readable report is in Chinese, which is fine for the author but creates an accessibility barrier for the dual-review agents (prompted in English) and for international collaborators; the M5 skill policy (§3) explicitly says "all review prompts/bundles for M5 should be English-only" but doesn't address the paper itself.** | **Suggestion:** Produce an English-language version of the paper alongside the Chinese version (or make English the primary, Chinese secondary) and add a language field to the evidence index so the gate can verify review language matches document language.

## Idea Generation

- **Severity: H** | **Component: idea-core** | **Location: no file — missing component** | **Problem:** The ecosystem has no visible mechanism for generating new research ideas or hypotheses; the "idea-core" name implies idea generation but the entire implementation is about running, reviewing, and gating a pre-specified pipeline — there is no exploration, hypothesis ranking, or novelty-seeking loop.** | **Suggestion:** Add an `idea_candidates.json` artifact schema with fields `{hypothesis, method, expected_observable, novelty_score, feasibility, status}` and a gate that requires at least N candidate ideas to be evaluated (even if rejected) before a project proceeds past M0.2.

## Risks

- **Severity: H** | **Component: workflow** | **Location: `m0.5-final-bounds.json`, `scope_note`** | **Problem:** The final bounds JSON contains a `scope_note` admitting the model is a "toy bootstrap ansatz, not a publishable dispersive analysis," yet the gate passes with `status: ok` — if this ecosystem is later used for real research, the same gate structure would pass physically meaningless bounds without warning.** | **Suggestion:** Add a `scope` enum to the gate schema (`"ecosystem_validation" | "preliminary_physics" | "publication_ready"`) and require the gate to emit a warning (or require explicit human approval) if scope is not `publication_ready` but bounds are being cited.

- **Severity: H** | **Component: compute-plane** | **Location: `m0.5-final-bounds.json`** | **Problem:** The D-term bound `[-14.98, -0.005]` spans three orders of magnitude and the radius bound `[0.007, 0.32]` fm² spans 1.5 orders of magnitude — these are uninformative envelopes that would trivially contain any reasonable physical value, raising the risk that the ecosystem is validating its ability to produce *any* number rather than *constraining* numbers.** | **Suggestion:** Add a literature-comparison gate: require the final bounds to be compared against at least one published value (e.g., lattice QCD `D^π(0) ≈ -1` from LHPC, or `r_g² ~ 0.3` fm² from dispersive analyses) and record whether the bounds are consistent, tighter, or vacuously wide.

- **Severity: M** | **Component: workflow** | **Location: all pipeline.json, `board_sync.commands`** | **Problem:** Board sync commands contain hardcoded GitHub Project item IDs and field IDs that will silently fail or sync to wrong items if the project is forked, items are deleted, or IDs change — no validation of the sync target exists.** | **Suggestion:** Add a pre-sync verification step that queries the item title/status before editing and records a `sync_verification` artifact; gate rejects if verification fails.

- **Severity: M** | **Component: workflow** | **Location: `m5-skill-policy.md` §4** | **Problem:** The "clean end-to-end replay" is defined but there is no evidence in the audit packet that it was ever executed; the M0.5 gate does not require it.** | **Suggestion:** Add `clean_replay_evidence` as a required field in the M0.5 gate schema, containing the commit SHA, `git status` output, and CI log path.

- **Severity: M** | **Component: hepar** | **Location: `pipeline.json` files, `review.claude_runner` and `review.gemini_runner`** | **Problem:** Review runner paths are absolute local paths (`/Users/fkg/.codex/skills/...`) that cannot be verified or reproduced; the M0.3 pipeline even uses a *different* gemini runner path (`tools/run_gemini_strict.sh`) than M0.4/M0.5, with no documented reason.** | **Suggestion:** Vendor runner scripts into the repo, add SHA-256 checksums to `meta.json`, and add a gate check that runner checksums match the expected values.

- **Severity: L** | **Component: compute-plane** | **Location: `PionGFFBootstrap.jl`, use of `MersenneTwister`** | **Problem:** MersenneTwister is deterministic but not cryptographically secure and its implementation can vary across Julia versions; the provenance records Julia version but not the RNG algorithm explicitly.** | **Suggestion:** Record `rng_type: "MersenneTwister"` in results provenance and add a note that reproducibility is only guaranteed within the same Julia major version.

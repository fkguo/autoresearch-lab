`idempotency_key` 重新调用 `search.step`。\n\n## 3. 核心抽象：Operator + SearchPolicy + Distributor（可替换）\n\n> 把“科学发现路径”落到可执行机制：Operator 是“如何变异/扩展 idea”；SearchPolicy 是“如何在树/种群中调度探索”；Distributor 是“预算如何分配到不同 LLM 与不同 operator/island”。\n\n> 增补（2026-02-12）：为实现“多 agents 物理学家社区/组团探索”，我们引入 **Team/Role 作为可插拔运行时拓扑**：同一个 island/方向不再对应单一生成器，而是对应一个由角色组成的团队（并行/串行可配置），其输出通过 artifacts 汇聚回 IdeaStore。\n\n### 3.1 `Operator`（发现算子）\n\n**职责**：输入一个 seed/idea-node，输出若干候选（带 rationale 草稿），并附带操作痕迹（trace）。\n\n建议最小接口（概念）：\n- `operator_id` / `operator_family`\n- `apply(input, context) -> [RationaleDraft]`\n- `operator_trace`：包含策略参数、随机种子、引用的证据 URI（用于审计/复现/回放）\n\n**Operator families（v0.2 建议）**  \n（把科学史/科学哲学转成可执行算子族；HEP pack 可先实现其中 4–6 个）\n\n- `AnomalyAbduction`：反常 → 解释（Kuhn/Peirce abduction）\n- `AssumptionInversion`：假设反转（Popper；增强可证伪性）\n- `SymmetryOperator`：对称性操作（破缺/恢复/推广/对偶）\n- `LimitExplorer`：极限外推（强/弱耦合、能标极限、维数极限）\n- `CrossDomainAnalogy`：结构/方法/现象映射（强制输出 mapping table；跨学科/跨分支方法迁移的一等公民）\n- `ProblemReduction`：问题抽象→归约到数学/CS 标准问题（优化/推断/图/方程/几何…）→ 检索既有解法并生成 transfer plan（重点覆盖“物理里难，但数学里已成熟”的场景）\n- `CombinatorialSynthesis`：方法×理论模块组合（IdeaSearch 风格再组合）\n- `ProtectiveBeltPatch`：Lakatos（保持 hard core，改 protective belt）\n- `RepresentationShift`：改变表述（变量替换、对偶变换、规范选择）\n\n### 3.2 `SearchPolicy`（搜索策略）\n\n**职责**：在预算约束下，决定“扩展哪个节点/哪个岛、使用哪个 operator、保留哪些候选”。\n\nv0.x 推荐先支持：\n\n1. **Divergent–Convergent 外环**（管线纪律）\n2. **Multi-Island Evolution 内核**（IdeaSearch 风格：不同策略群体并行 + repopulate）\n3. 可选：**BFTS / BeamSearch**（对树结构分支的局部 best-first）\n\n> 设计要点：SearchPolicy 只依赖 `IdeaNode` 与 `Evaluator` 的输出，不依赖领域细节。\n\n#### 3.2.1 Multi-Island 状态机（最小可实现规格）\n\n为避免“同名多岛”但实现不可互换，v0.2 要求至少声明以下状态与迁移：\n\n- **States**：`SEEDING → EXPLORING → CONVERGING → STAGNANT → REPOPULATED | EXHAUSTED`\n- **Triggers（示例）**：\n  - `SEEDING → EXPLORING`：population 达到 `min_pop_size`\n  - `EXPLORING → CONVERGING`：top-k 语义多样性低于阈值（避免模式坍缩）\n  - `EXPLORING/CONVERGING → STAGNANT`：`best_score_improvement(last_n_steps) < ε`\n  - `STAGNANT → REPOPULATED`：执行 repopulate（从 donor islands 注入 migrants）\n  - `STAGNANT → EXHAUSTED`：预算不足以完成一次最小 step（`budget_remaining < min_step_cost`）\n  - `REPOPULATED → EXPLORING`：重置 stagnation 计数器后继续\n\n需要对外暴露可替换的谓词/策略接口（概念）：\n- `should_repopulate(island_state, history, budget_remaining) -> bool`\n- `migrate(donor_islands, target_island, n_migrants, selection_fn) -> IdeaNode[]`\n\n### 3.3 `Distributor`（预算分配/调度器）\n\n**职责**：把生成请求分配给：\n\n- 不同 LLM backends（异构模型表型互补：explorer vs converger vs critic）\n- 不同 islands / operator families\n\nv0.2 可先引入 **softmax+EMA** 作为 baseline（IdeaSearchFitter 给出可复用机制）：\n\n- 为每个 backend/operator 维护近期 reward 的 EMA\n- 用 `softmax(score/T)` 分配生成配额（避免单点收敛）；更具体地，可采用：\n  - `p_i = exp(s_i / T_model) / Σ_j exp(s_j / T_model)`  \n  - 其中 `s_i` 为后端/算子的 reward EMA，`T_model` 控制探索-利用权衡\n\n> 但 softmax+EMA 不显式建模不确定性/方差、冷启动、非平稳漂移与成本约束；用于异构 LLM 编排时往往需要更强的 allocator。\n> 建议将 Distributor 设计为 **BanditPolicy 可插拔**，并提供可回放的 decision trace（见 `docs/plans/2026-02-12-bandit-distributor-alternatives.md`；统计物理视角与可审计日志建议见 `docs/plans/2026-02-12-statphys-distributor-policies.md`）。\n\nv0.3+ 推荐默认升级为 **cost-aware + non-stationary bandit**：\n\n- **不确定性/方差**：UCB 系列（如 variance-aware bonus）或 Bayesian（TS/Bayes-UCB）\n- **非平稳**：discount / sliding-window 更新；必要时 change-point reset\n- **成本敏感**：以 `$ / tokens / wall_clock_s / team_cost` 等资源为约束或罚项（BwK / Lagrangian）\n- **可审计/可复现**：deterministic mode（或固定 seed + 记录随机数）+ ledger 事件（per-step 的 per-arm stats 与 index 分解）\n\n**Budget Circuit Breaker（全局熔断器，必须）**：\n- 当 `total_cost > max_budget` 或 `total_tokens > max_tokens` 或 `wall_clock_s > max_wall_clock_s` 时，立即终止所有 pending 生成/评估，进入 `EarlyStop`，并写入 `budget_checkpoint` ledger 事件。\n- 建议提供 `degradation_order[]`：当预算紧张时的降级顺序（例如先减少评估轮数，再减少 islands，再禁用跨域算子）。\n\nreward 信号分两类：\n- **短期 proxy**：grounding ratio、novelty proxy、结构化可编译率（是否能形成 Canonical IdeaCard）\n- **长期 outcome**：A0.2 通过率、C2 成功率、W_compute 成功率（延迟奖励）\n\n**Phenotype profiling（用于异构分工与调度，不是“模型排行榜”）**：\n- 迁移自 IdeaSearchFitter 的动态过程指标：`Iter@k`（不同 epoch/step 的累积成功）、epoch 分布（中位数/3-4 分位）、成本（word/token）。\n- ideation 侧建议最小映射：`A0_pass@k`、`epochs_to_first_grounded_claim`、`tokens_per_promoted_node`、`checker_disagreement_rate`。这些指标应写入 `origin`/`eval_info` 以便回放与调度学习。\n\n### 3.4 Physicist Community：Team/Role 组团探索（OpenClaw Broadcast Groups 启发）\n\n> 目的：把“多 agent 协作”从泛泛口号变成可配置、可审计、可扩展的运行时拓扑；同时避免把评审/审查逻辑硬编码进某个单体提示词。\n\n#### 3.4.1 核心概念（最小可实现）\n\n- **Role**：一个“具名职责 + 工具/模型权限 + 输出契约”的执行单元（可由不同 LLM 或工具沙箱承载）。\n- **Team**：一组 Roles，围绕同一研究方向/策略 island 协作；Team 有协调策略（并行/串行/分阶段）。\n- **Community**：多个 Teams 的集合 + 中央仲裁/汇总角色（用于跨团队排名、repopulate、冲突裁定）。\n\n建议把 multi-island 解释为：`island_id` 同时标识\n1) 搜索策略群体（operator/constraints 权重配置）与\n2) 该策略群体对应的 Team（role composition + coordination policy）。\n\n#### 3.4.2 角色建议（HEP-first 的最小社区）\n\n> v0.2 不要求一次性全实现；但需要把“角色接口”设计出来，后续可逐步替换为更强的执行体。\n\n- `Ideator`（发展想法）：发散生成、提出机制与测试点（输出 `RationaleDraft`）。\n- `Librarian`（证据检索）：INSPIRE/PDG/讲义/综述检索，产出 `evidence_packet`（URI + 摘要 + 相关性）。\n- `Formalizer`（结构化形式化）：把 rationale 收敛为 `IdeaCard`（schema-validated）。\n- `Derivation`（推导/一致性检查）：对称性/量纲/极限/一致性快速检查，产出 `consistency_report`（可带最小手算/公式）。\n- `Coder`（最小计算原型）：把 compute plan 落到可执行草图（最小脚本/伪代码/数值检查路径）。\n- `Checker`（复核/复现）：独立重算/重查（clean-room），给出一致/不一致结论与最小反例。\n- `Referee`（可行性/新颖性/影响审查）：按 rubric 打分，但必须输出“**创新增量**”而非修辞差异（见 6.2）。\n- `Editor`（汇总/归档）：把各角色结构化输出合并为 `IdeaNode` 更新（trace 完整、可回放）。\n\n> Packaging 建议：`Referee` 不建议作为“与 idea-generator 并列的独立产品 agent”。更合理的做法是：把它当作 **idea-generator 的可插拔 role**（可由独立 session/独立模型承载，以获得 clean-room 效果与更严格的工具权限），从而同时满足“模块化替换”和“端到端门禁一致性”。\n\n#### 3.4.3 协调策略（并行 vs 串行 vs 分阶段）\n\n三种基本模式（可作为 `TeamPolicy` 的枚举）：\n\n1. **Parallel Divergence**：并行广播给多个 roles（尤其是多个 ideator/referee），提高多样性与盲点覆盖。\n2. **Sequential Convergence**：`Librarian → Ideator → Formalizer → Referee`，用证据与结构化输出抑制幻觉与格式漂移。\n3. **Stage-Gated**：在关键门禁前强制进入特定阶段（如 promote 前必须跑 `Checker` 与 `GroundingAudit`）。\n\n> 关键纪律：默认 **Role clean-room**；role 之间不共享草稿，只共享输入上下文包与最终 artifacts。若触发辩论，则以 `debate_packet`（point/counterpoint + evidence_uris）形式显式发生。\n\n> **成本感知约束**：Team topology 的 token 消耗近似为 `Σ(roles) × per-role-cost`；Distributor 在分配预算时必须将 team composition 纳入每步成本估算（而非假设每步 = 1 次 LLM 调用）。`BudgetEnvelope.extensions` 可承载 `team_cost_multiplier`/`role_cost_table` 等运行时参数。\n\n---\n\n## 4. 强制两阶段：RationaleDraft → Canonical IdeaCard（Explain-Then-Formalize）\n\n### 4.1 为什么必须强制\n\nExplain-Then-Formalize 的价值在于：\n\n- 允许 **发散**（类比、隐喻、反转假设）  \n- 但必须在进入门禁/下游前完成 **形式化**（可验证字段齐全）\n\n### 4.2 两阶段产物（建议）\n\n1. `RationaleDraft`（允许高温）：\n   - WHY：动机/反常/类比映射表/机制猜想\n   - 风险：潜在已知/folklore、物理一致性风险、验证优先级\n   - 下一步最小验证：1–3 个 kill criteria\n\n2. `IdeaCard`（低温约束 + schema 校验）：\n   - `thesis_statement`\n   - `testable_hypotheses[]`\n   - `required_observables[]`\n   - `candidate_formalisms[]`（必须映射到 DomainPack 的 formalism registry；否则不得 promote 到 C2）\n   - `minimal_compute_plan[]`\n   - `claims[]`（claim-level 溯源；每条 claim 至少包含 support_type + evidence_uris；对 `llm_inference/assumption` 必须给 verification_plan；可选 verification_status）\n\n> 硬规则：任何 idea 进入 Ranking / A0.2 前，必须完成 `IdeaCard` 生成与 schema 验证。\n\n#### 4.2.1 Grounding Audit Gate（IdeaCard 生效前的强制门禁）\n\n为避免“看似有引用但引用并不支撑 claim”的失真 provenance，IdeaCard 只有在通过 grounding audit 后才被视为“有效可推广对象”：\n\n1. **URI 可解析（active resolution）**：`claims[].evidence_uris[]` 必须通过 **active lookup** 验证可解析（INSPIRE API / DOI resolver / KB artifact existence check 等），不得仅做格式校验；不存在 phantom 引用。解析失败必须写入 `IdeaNode.grounding_audit.failures[]`，并将 `status` 置为 `fail` 或 `partial`。\n2. **数据一致性**：`support_type=data` 的数值类 claim，必须与 PDG/HEPData 在约定容差内一致（否则标记 FAIL）。\n3. **推断透明**：`support_type=llm_inference/assumption` 必须有 `verification_plan`（至少 1 条 kill criterion）。\n4. **folklore 预筛**：产出 `folklore_risk_score ∈ [0,1]`；超过阈值则必须走 `A0-folklore` 人类裁定。\n5. **晋升门禁（强约束）**：`node.promote` 的成功条件必须包含 `grounding_audit.status == pass`。`partial/fail` 一律阻塞晋升（返回 `grounding_audit_failed`），避免“带着缺口进入 C2”。\n\n审计输出写入 `IdeaNode.grounding_audit`（并可汇总进 `idea_evidence_graph_v1.json`）。Grounding Audit 的默认触发点应当是 `eval.run`（当 `EvaluatorConfig.dimensions` 包含 `grounding` 时）；也允许由 `Checker` 角色在生成后即时执行（以加速剪枝）。\n\n---\n\n## 5. agent）\n\n维度建议延续：\n- novelty / feasibility / impact / tractability / grounding\n\n关键工程约束：\n- evaluator 返回不只是分数，还要返回 **可再投喂的诊断**（参考 IdeaSearch-framework 的 score+info）\n- 多 agent 评审默认 **clean-room**（互不共享对话记忆），直到触发“结构化辩论”才允许受控信息流\n\n#### 6.1.1 clean-room 与结构化辩论（最小协议）\n\n- clean-room：每个 evaluator 在独立会话/上下文中完成评分与证据检索（不共享草稿、不中途互相影响）。\n- 触发辩论：任一维度分歧超过阈值（例如 `|Δscore| > 2`）或出现互斥结论（“已发表/未发表”）。\n- 辩论输出必须结构化：point/counterpoint + evidence_uris + 最终裁定理由（作为 `eval_info` 的一部分写回）。\n\n### 6.2 Novelty：四层栈 + folklore 风险\n\n沿用 2026-02-11 的四层 novelty pipeline，但把 folklore 风险变成显式字段：\n- `folklore_risk`（高则必须走 `A0-folklore` 人类裁定）\n\n**重要补充：避免把“细枝末节”当成创新（Referee 硬约束）**\n\n当 `EvaluatorConfig.dimensions` 包含 `novelty` 时，负责 novelty 维度的 Referee 输出必须包含一个 `novelty_delta_table`（可作为 `eval_info` 的结构化字段或独立 artifact），至少回答：\n\n1. **closest prior**：最相近的 1–3 篇/方向（URI + 一句摘要）\n2. **delta type**（枚举，可多选）：`new_mechanism | new_observable | new_regime | new_method | new_formalism | new_dataset | new_constraint`\n3. **delta statement**：每个 delta 必须是一句“可被证伪”的陈述（不是措辞变化）\n4. **non-novelty flags**：显式标注以下情况不计为主要创新：参数微调、符号替换但等价、仅换叙述不换预测、仅合并已知组件但无新可检验结论\n5. **verification hook**：每个 delta 关联一个最小验证（可文献/可计算/可观测）\n\n### 6.3 Ranking：Pareto + Tournament（Elo）\n\n- Pareto：多目标前沿保证不被单一分数绑架\n- Tournament/Elo：降低评分尺度漂移影响，促进相对比较的稳定性\n  - **硬约束**：Elo 必须是 **bounded + deterministic** 的（否则成本不可控/不可复现）。建议 `rank.compute(method=elo)` 接受 `elo_config={max_rounds, seed}`，并把 pairing 规则写入 ranking artifact 以便回放。\n\n---\n\n## 7. DomainPack（领域插件）最小化设计（避免过度抽象）\n\nv0.2 建议把扩展点收敛为 6 类（HEP 先硬编码实现，再抽取接口）：\n\n1. `seed_sources`\n2. `operators`（或 operator 参数化/模板）\n3. `constraints_and_validators`\n4. `retrieval_recipes`（INSPIRE 查询模板、关键词扩展、分类映射）\n5. `feasibility_estimators`\n6. `method_compilers`（IdeaCard → C2 handoff；必须可校验）\n\n> DomainPack 的“知识载体”主要是：ontology + prompt templates + validators，而不是把物理写成大量 Python 规则（可维护性更好）。\n\n**重要：HEP-first ≠ HEP-only（跨学科方法迁移必须预留接口）**  \n许多突破来自“方法/表述/直觉”的跨域迁移，而不是同一子领域内的微调。因此在架构层必须保证：\n- `domain` 只是 campaign 的 **primary target domain**（用于验收/约束/评估权重），不应把检索与算子空间硬锁死在 HEP。\n- 支持“可插拔的 MethodPacks / CrossDomainPacks”：它们提供跨学科的 `operators + retrieval_recipes + validators`（例如统计物理/信息论/数学工具/ML），由 `TeamPolicy`/预算降级策略显式启用/禁用（见 `disable_cross_domain_operators`）。\n- 跨域算子必须产出显式 `mapping table + invariants + kill criteria`（避免隐喻式类比污染），并通过 grounding audit（引用可解析）与目标域硬约束（量纲/对称性/可证伪性）。\n- **“数学已解决”优先发现**：当目标域瓶颈可被抽象/归约为数学或其他学科的标准问题时（如优化、统计推断、图算法、偏微分方程、几何/拓扑对象），系统应优先启用 `ProblemReduction`/`TechniqueTransplant` 类算子：先给出抽象对象与等价/近似归约，再检索该对象在数学/CS 的经典解法与已知边界条件；仅在 transfer plan 通过 toy check/硬约束后才允许进入晋升/排名。\n\n**formalism registry（为 C2 提供可校验映射）**：\n- DomainPack 必须声明 `formalism_id → {c2_schema_ref, validator, compiler}` 的映射。\n- `candidate_formalisms[]` 必须来自该 registry；否则 `node.promote` 必须失败（schema_validation_failed 或 grounding_audit_failed 或 formalism_not_in_registry）。\n\n---\n\n## 8. 与现有生态圈集成（HEP-Autoresearch）\n\n### 8.1 输入（seed sources）\n\n- C1 gaps（系统性缺口）\n- KB priors（已有笔记/失败记录/方法痕迹）\n- PDG/HEPData tensions（反常/张力）\n- user OperatorSpec：最小可执行契约（建议）\n\n这不是最终 schema（避免过早冻结），但建议 `HEP DomainPack` 内部先用同形对象描述算子，后续再升级为稳定 schema：\n\n```text\nOperatorSpec:\n  operator_id: string                      # 稳定 ID（用于 provenance）\n  family: enum                             # 算子族（见下）\n  seed_kinds: [enum]                       # 允许的输入 seed 类型\n  requires_roles: [RoleId]                 # 推荐/必需角色（Librarian/Derivation/…）\n  hard_constraints: [ConstraintId]         # 必须执行的 validators（量纲/对称性/引用可解析…）\n  output_kind: \"RationaleDraftV1\"          # v0.x 固定\n  trace_schema: object                     # operator_trace 的最小字段集（可审计）\n  prompt_templates: {prologue, examples, epilogue, extractor?}\n  knobs: {temperature, max_variants, ...}  # 可调参数（进化/多岛可学习）\n```\n\n**硬纪律**：\n- 任意 `Operator.apply()` 生成的候选必须带 `operator_trace`（参数/随机种子/引用证据 URI/父代节点）。\n- `Operator` 只负责“生成与局部变异”；是否保留/晋升由 `SearchPolicy + Evaluator + Gates` 决定。\n\n---\n\n## 2. 可执行算子族（从科学哲学/历史抽取 → 直接可落地）\n\n每个算子族都给出：**输入 seed → 输出约束 → 必做验证 → 常见误判**。HEP-first 的 MVP 建议先实现其中 4–6 个（用 Multi-Island 保多样性）。\n\n### 2.1 `AnomalyAbduction`（反常→解释；Peirce/Kuhn 风格）\n\n- **Seed**：张力/反常（PDG tension、实验异常、理论不自洽边界条件）\n- **输出必须包含**：\n  - 最小“解释机制”候选（不超过 3 条）\n  - 每条机制的 *kill criteria*（1–3 条可证伪条件）\n  - 最小证据计划：哪些观测量/哪些综述/哪些关键测量能否定或支持\n- **必做验证**：引用可解析（grounding）、“解释对象”是否被准确表述（不要造假反常）\n- **常见误判**：把“尚未复现/系统误差”当成新物理；或把“模型换个名”当机制增量\n\n适配 Team/Role：`Librarian → Ideator → Referee → Checker`\n\n---\n\n### 2.2 `AssumptionInversion`（假设反转；Popper 的可证伪性杠杆）\n\n- **Seed**：某个默认假设（locality、naturalness、flavor universality、thermal history…）\n- **输出必须包含**：\n  - 被反转的假设（显式列出）\n  - 反转后产生的“新可检验预测”（≥1）\n  - 新增自由度/代价（复杂度/可行性）与最小化路径\n- **必做验证**：新预测必须能映射到 `IdeaCard.required_observables[]`\n- **常见误判**：反转假设但没有任何新预测；或预测不可触达（不可证伪）\n\n适配 Team/Role：`Ideator → Formalizer → Referee`\n\n---\n\n### 2.3 `ProtectiveBeltPatch`（Lakatos：保持 hard core，改辅助带）\n\n- **Seed**：一个研究纲领（hard core + auxiliary hypotheses），或一个已知方案的失败点\n- **输出必须包含**：\n  - hard core（不可动的核心承诺）列表\n  - protective belt（允许改动）列表 + 改动提案\n  - “为何这不是 ad hoc”：至少给出一个独立可检验后果\n- **必做验证**：对已知成功现象的回归约束（不破坏已知结果）\n- **常见误判**：为适配一个点而引入不可控自由度；或者补丁无法产生新预测\n\n适配 Team/Role：`Librarian → Ideator → Derivation → Referee`\n\n---\n\n### 2.4 `SymmetryOperator`（对称性操作：破缺/恢复/推广/对偶）\n\n- **Seed**：某个对称性结构或其破缺模式（global/gauge、flavor、C/P/T、SUSY、dual symmetry…）\n- **输出必须包含**：\n  - 操作类型：`break | restore | extend | gauge | dualize | anomaly_match`\n  - 新的选择规则/禁戒/允许通道（至少一条可检验）\n  - 与已有约束的冲突点（潜在 kill criteria）\n- **必做验证**：量纲一致性、对称性一致性（最小推导/极限检查）\n- **常见误判**：只改“表述”不改物理；或忽略 anomaly/consistency\n\n适配 Team/Role：`Ideator → Derivation → Checker → Referee`\n\n---\n\n### 2.5 `LimitExplorer`（极限与能标：强/弱耦合、IR/UV、large-N、维数延拓）\n\n- **Seed**：某个理论/模型/方法在极限下的行为或已知可解角落\n- **输出必须包含**：\n  - 选择的极限（参数→0/∞、能标→IR/UV）\n  - 极限下可计算的量与对全域的外推假说\n  - 极限之间的插值/匹配计划（最小 compute plan）\n- **必做验证**：极限可计算性（至少给出一个可执行/可手算的 check）\n- **常见误判**：把“极限下成立”误推广到一般点；或没有任何可验证插值路径\n\n适配 Team/Role：`Derivation → Coder → Checker`\n\n---\n\n### 2.6 `RepresentationShift`（表述变换：变量/规范/基底/对偶表述）\n\n- **Seed**：当前表述导致的困难（非局域、强耦合、非线性、发散结构…）\n- **输出必须包含**：\n  - 变换类型：变量替换/规范选择/基底旋转/对偶表示\n  - 变换后“更容易计算/更透明的量”（至少一项）\n  - 不变性声明（哪些量必须保持不变）\n- **必做验证**：同一物理量在两表述下的一致性 check（toy check 也可）\n- **常见误判**：把重写当创新；没有任何新可计算/新可检验后果\n\n适配 Team/Role：`Derivation → Formalizer → Referee`\n\n---\n\n### 2.7 `CrossDomainAnalogy`（结构/方法/现象类比；要求显式 mapping table）\n\n- **Seed**：目标域问题的“抽象结构”（弱信号探测/多尺度/网络效应/相变/噪声抑制…）\n- **输出必须包含**：\n  - 源域→目标域 mapping table（至少 5 个对应项）\n  - 迁移对象类型：`structure | method | phenomenon`\n  - 目标域/源域声明：`target_domain`（用于验收与硬约束）+ `source_domain`（用于检索与类比的知识域）\n  - 目标域硬约束（守恒/对称性/量纲/实验可得性）列表\n  - invariants（不变性）清单：迁移过程中哪些量/结构必须保持（否则类比无效）\n- **必做验证**：mapping table 与约束一致性（不满足直接拒绝）\n- **常见误判**：类比停留在隐喻；或者引入违反物理约束的“好点子”\n\n适配 Team/Role：`Librarian → Ideator → Derivation → Referee`\n\n> 备注（跨学科的一等公民）：  \n> - `CrossDomainAnalogy` 不只是“从别的物理分支借直觉”，也包括 **跨学科借方法**（例如信息论/统计学习/优化/数值分析/网络科学）。  \n> - 对“method transfer”必须强制输出：源方法的最小可复现描述（含引用）+ 在目标域的 toy check（哪怕是最小极限/维度分析/一致性检验），否则一律视为“隐喻”并拒绝晋升。\n\n---\n\n### 2.8 `CombinatorialSynthesis`（模块组合：把“可复用组件”当基因片段）\n\n这是对 IdeaSearch/进化框架的“物理研究计划版”迁移：\n\n- **Seed**：组件库（方法模块/理论模块/观测量/数据源/近似技术）\n- **输出必须包含**：\n  - 组合清单（模块 A×B×C）\n  - 组合产生的“新可检验结论”（而不是模块并列）\n  - 冲突/不兼容点与修复路径\n- **必做验证**：最小一致性（量纲/对称性/极限）+ novelty delta table（避免“拼装即创新”）\n- **常见误判**：纯拼装无新预测；或只产生工程差异不产生科学差异\n\n适配 Team/Role：`Ideator → Formalizer → Referee → Checker`\n\n---\n\n### 2.9 `ConjectureGeneralization`（从例子到猜想：模式提炼→可证伪外推）\n\n- **Seed**：已知一组可计算例子/已知 theorem/已知分类\n- **输出必须包含**：\n  - 猜想的精确定义（条件/结论）\n  - 至少一个潜在反例搜索策略（kill criteria）\n  - 最小验证集（多例子 cross-check）\n- **必做验证**：反例优先（由 `Checker/Referee` 角色尝试击杀）\n- **常见误判**：过拟合少数例子；没有反例/边界条件意识\n\n适配 Team/Role：`Derivation → Checker → Referee`\n\n---\n\n### 2.10 `FalsificationPressure`（反证压力测试：专门生成“击杀路径”）\n\n这是把 Popper 的精神变成可执行算子：**不是产生新 idea，而是产生杀手测试**。\n\n- **Seed**：已有候选 `IdeaNode`（通常来自其他算子）\n- **输出必须包含**：\n  - 1–5 条 kill criteria（越短越好）\n  - 每条 kill criteria 的证据来源或可执行检查路径\n  - 若 kill criteria 不成立，说明“如何修复”（形成可迭代的 fix suggestion）\n- **必做验证**：grounding（kill criteria 的依据必须可追溯）\n- **常见误判**：泛泛而谈“需要更多数据”；或提出不可执行的反证\n\n适配 Team/Role：`Checker → Referee`（clean-room 默认）\n\n---\n\n### 2.11 `TechniqueTransplant`（跨学科方法移植：把“方法”当可执行资产）\n\n这是把跨学科突破的常见路径（“方法迁移”而非“结果照搬”）做成显式算子：从源学科挑选一项可复用方法，并在目标域给出**可执行的最小落地**。\n\n- **Seed**：目标域的瓶颈（不可计算/不可辨识/噪声强/维度高/参数不可约…）+ 允许的约束（预算/可解释性/可证伪性）\n- **输出必须包含**：\n  - `source_method`：方法名称 + 一句话核心机制 + ≥1 条一手引用（文献/书/讲义 URI）\n  - `transfer_plan`：迁移步骤（如何把输入/输出/损失/近似对应到目标域）\n  - `compatibility_checks`：至少 2 条“必须过”的一致性检查（量纲/极限/对称性/可解释性/可复现性）\n  - `minimal_toy_check`：一个最小可执行验证（toy model/极限/已知结果回归），用于快速击杀“看似有效但不落地”的移植\n  - `kill_criteria`：若方法不适用，最短的否决条件（避免陷入工程细节）\n- **必做验证**：引用可解析（grounding）+ toy check 可执行（由 `Coder/Checker` 角色验证）\n- **常见误判**：把“方法名”当迁移；没有具体 transfer plan；或方法只是在工程上更方便但不产生新可检验结论\n\n适配 Team/Role：`Librarian → Ideator → Coder → Checker → Referee`\n\n---\n\n### 2.12 `ProblemReduction`（问题抽象与归约：优先挖掘“数学/CS 已成熟解法”）\n\n很多物理研究的“难点”并不在物理本体，而是隐藏在一个更通用的抽象问题里（优化、统计推断、图算法、PDE、几何/拓扑对象、数值稳定性等）。一旦归约成立，常常能直接借用数学/CS 的成熟工具链。\n\n- **Seed**：目标域瓶颈 + 最小形式化信息（变量/约束/目标/可观测量）+ 明确允许的近似（若有）\n- **输出必须包含**：\n  - `abstract_problem`：抽象问题类型（例如 `optimization | inference | graph | pde | geometry | algebra | control | signal_processing | numerics`）\n  - `reduction_map`：目标域 → 抽象域的映射表（变量/目标/约束/对称性/不变性；至少 8 个对应项）\n  - `assumptions_and_limits`：归约成立的条件与边界（哪些近似是必需的；失败会如何显现）\n  - `known_solutions`：≥2 个候选成熟解法（每个包含：前提条件、复杂度/代价、典型 failure modes、≥1 条一手引用 URI）\n docs/plans/2026-02-12-bandit-distributor-alternatives.md ---\n\n# Better alternatives to Softmax+EMA for Distributor budget allocation (MAB)\n\n> Scope: allocating budget across multiple **LLM backends and/or operators** (multi-armed bandit) under: scale, uncertainty/variance, cold start, non-stationarity, and cost sensitivity.\n>\n> Goal: replace (or at least **strictly dominate**) “softmax(score-EMA)” with policies that have **principled exploration**, **variance awareness**, and **explicit cost/budget constraints**, while staying **deterministic/auditable**.\n\n---\n\n## 0) Why Softmax + EMA is brittle (and where it breaks)\n\nSoftmax-on-EMA is basically *Boltzmann exploration with ad‑hoc recency weighting*. It’s attractive because it’s simple, but it fails precisely where LLM orchestration is hard:\n\n- **No uncertainty modeling**: it treats an EMA mean as truth; high-variance arms can look “best” by luck and steal budget.\n- **Scale pathologies**: with many arms, softmax probabilities can become numerically tiny; exploration becomes accidental and can “die out”.\n- **Cold start**: new arms start with no data; unless you hand-tune priors/initial scores, they get starved (or over-fed) in unstable ways.\n- **Non-stationarity**: EMA is a blunt forgetting mechanism; it cannot *detect* changes, only smear them.\n- **Cost blindness**: if arms differ by $/token, latency, or multi-role team cost, “reward-only” softmax can overspend.\n\nTreat Softmax+EMA as a **baseline** (easy to ship), not a “final” allocator.\n\n---\n\n## 1) Canonical bandit alternatives (by failure mode)\n\n### A) Uncertainty + variance (stochastic, stationary)\n\n**UCB family (optimism under uncertainty)**\n- **UCB1** (Auer, Cesa‑Bianchi, Fischer, 2002): pick `argmax_i (mean_i + bonus_i)`; bonus shrinks as `n_i` grows; deterministic, simple, strong guarantees.\n- **UCB‑V / empirical Bernstein UCB** (Audibert, Munos, Szepesvári, 2009): replaces Hoeffding-style bonuses with **variance-aware** bonuses → more stable when reward noise differs across backends/operators.\n- **KL‑UCB** (Garivier & Cappé, 2011): uses KL divergence confidence bounds; typically tighter than UCB1 for bounded rewards.\n\n**Bayesian / sampling**\n- **Thompson Sampling** (Thompson, 1933; modern analysis e.g. Agrawal & Goyal, 2012/2013): sample a plausible mean for each arm from its posterior and pick the best; handles uncertainty naturally and tends to behave well in practice.\n- **Bayes‑UCB** (Kaufmann, Cappé, Garivier, 2012): choose arm by a high posterior quantile (an “optimistic” posterior index). Often **deterministic given the posterior** (helpful for audit).\n\n**Design translation for LLM routing**\n- Use reward normalization into `[0,1]` (or bounded) so “bounded bandit” assumptions roughly hold.\n- Prefer **variance-aware UCB** when evaluator noise is heteroscedastic (common with LLM-judge scores).\n\n### B) Cold start + scale (many arms, shared structure)\n\nWhen the number of arms grows (many operators / model variants), per-arm learning becomes sample-inefficient. Standard fix: **structure**.\n\n**Contextual bandits (parameter sharing across arms)**\n- **LinUCB** (Li, Chu, Langford, Schapire, 2010) and **OFUL / linear bandits** (Abbasi‑Yadkori, Pál, Szepesvári, 2011): assume expected reward is (approximately) linear in features; generalize across arms and contexts.\n- **Linear Thompson Sampling** (Agrawal & Goyal, 2013): Bayesian linear bandit; strong cold-start behavior if you encode good priors/features.\n\n**Practical features for LLM Distributor**\n- Prompt/task features (length, domain tags, required tools).\n- “Operator phenotype” features (expected novelty vs grounding vs formalization rate).\n- Backend features (cost, latency, context length, tool support).\n\n**Hierarchical priors (arm sharing without explicit features)**\n- Use a population prior across arms (backend/operator family) so new arms inherit reasonable uncertainty and don’t require hard-coded initial scores.\n\n### C) Non-stationarity (drift, regime switches, “provider changes”)\n\nLLM routing is *non-stationary* (model updates, rate limits, safety filters, traffic spikes).\n\n**Forgetting / windowing**\n- **Discounted UCB** / **Sliding-Window UCB** (Garivier & Moulines, 2011): keep only recent evidence (window) or exponentially downweight the past (discount). Deterministic and easy to implement.\n\n**Adversarial-robust (worst-case drift)**\n- **EXP3** (Auer et al., 2002): multiplicative-weights bandit for adversarial rewards; maintains probabilities, but updates include importance weighting (higher variance).\n- Variants for drifting best-arm (**shifting regret**), e.g. EXP3.S / “tracking the best expert”, and **Rexp3** / variation-budget approaches (Besbes, Gur, Zeevi, 2014).\n\n**Change-point detection + reset (engineering-friendly)**\n- Run a lightweight detector (e.g., CUSUM/Page-Hinkley) per arm (or globally) and **reset** that arm’s statistics on detected shifts. This is often more interpretable than trying to tune a single EMA constant forever.\n\n### D) Cost-sensitive / budgeted (dollars, tokens, wall-clock, multi-role teams)\n\nIf each pull has a different cost, the correct objective is not “maximize mean reward”, but something like:\n\n- maximize **reward per unit cost** (ratio), or\n- maximize reward subject to **budget constraints** (knapsack), or\n- maximize `reward − λ·cost` (Lagrangian).\n\n**Bandits with Knapsacks (BwK)**\n- **BwK** (Badanidiyuru, Kleinberg, Slivkins, 2013; later extensions e.g. Agrawal & Devanur, 2016): bandits with *resource consumption* constraints; canonical framework when you have a total budget and per-action costs.\n\n**Engineering translation**\n- Treat each action as consuming a vector cost: `$`, `tokens`, `wall_clock_s`, and (optionally) `role_cost` (Team topology).\n- Use a primal-dual/Lagrangian view: maintain shadow prices for resources and choose actions that maximize `UCB(utility) − Σ_k λ_k·cost_k`.\n\n---\n\n## 2) Concise recommendation (what to implement for idea-generator)\n\n### Recommended default policy (v0.3 target): cost-aware non-stationary contextual bandit\n\n1) **Contextual** (for scale + cold start): LinUCB / OFUL *or* Linear Thompson Sampling using prompt/operator/backend features.\n2) **Non-stationary** (for drift): discounted or sliding-window updates of sufficient statistics;

Here is a comprehensive review of the Phase 1 Batch 2 Core Abstraction Layer packet, evaluated strictly against your defined criteria.

### Executive Summary
Overall, this is a highly pragmatic and well-designed batch. You have successfully avoided over-engineering by keeping registries static and separating state definitions from execution logic. The use of branded types (H-15a) and strict literal unions (H-03, H-04) provides excellent domain semantics. 

**Primary Area of Concern:** The legacy mapping table in `run-state.ts` (H-03) is **not invertible**, despite the evaluation criteria asking if it is. Many-to-one mappings inherently lose data. See the H-03 section for details and recommendations.

### File 1: `packages/shared/src/ecosystem-id.ts` (H-15a)

**Evaluation:**
*   **Domain Semantics:** Excellent. The use of a branded type (`__brand: 'EcosystemId'`) is the perfect pattern here. It prevents developers from accidentally passing a standard UUID or raw string into a function expecting a validated Ecosystem ID.
*   **Naming:** Prefixes (`proj`, `run`, `art`) are concise and standard.
*   **Contracts:** `ParsedEcosystemId` provides a clean, unambiguous boundary contract.
*   **Over-engineering:** None. This is the exact right amount of type safety for a foundational identifier.

**Technical Watch-out:**
*   **Parsing Logic:** Because the opaque part allows underscores (`[a-zA-Z0-9._-]{1,200}`), ensure `parseEcosystemId` splits on the **first** underscore only. A naive `raw.split('_')` will break if the opaque ID is `my_custom_id` (resulting in `proj_my_custom_id`). 
    *   *Recommendation:* Use a regex like `/^([a-z]+)_(.+)$/` or `indexOf('_')` for parsing.

### File 2: `packages/shared/src/artifact-ref.ts` (H-18)

**Evaluation:**
*   **Domain Semantics:** Clear distinction between the lightweight URI reference (`RunArtifactRef`) and the strict, content-addressed schema (`ArtifactRefV1`). 
*   **Naming:** The `hep://` URI scheme is a great, unambiguous way to route internal resources.
*   **Contracts:** Re-exporting `RunArtifactRef` for backward compatibility with `hep-mcp` ensures a smooth cross-component boundary.
*   **Over-engineering:** Keeping the execution/fetching logic out of this file keeps the abstraction pure.

**Technical Watch-out:**
*   **URI Encoding:** Artifact names often contain spaces, slashes, or special characters (e.g., `my report v2.pdf`). 
    *   *Recommendation:* Ensure `makeRunArtifactUri` uses `encodeURIComponent(artifactName)` and `parseHepArtifactUri` uses `decodeURIComponent`. The test coverage mentions encoding, which is a good sign, but it is critical for the boundary contract.

### File 3: `packages/shared/src/run-state.ts` (H-03)

**Evaluation:**
*   **Domain Semantics:** Separating Run-level states (`running`) from Step-level states (`in_progress`) is a brilliant semantic distinction. It prevents logical bugs where a step's state is accidentally applied to the whole run.
*   **Naming:** Consistent `snake_case` throughout.
*   **Contracts:** `isTerminalRunState` and `isActiveRunState` provide excellent, reusable boolean checks for orchestrators.

**Critical Feedback on Criterion 4 (Mapping tables: complete, invertible?):**
*   **Completeness:** Yes, it covers a wide array of legacy states.
*   **Invertibility:** **No, it is not invertible.** You have a many-to-one mapping. 
    *   Example: `done ← done, completed, DONE, early_stopped, skipped, abandoned`.
    *   If a legacy system sends `skipped`, it becomes `done`. If you try to invert this to send a status *back* to the legacy system, you only have `done`. You cannot programmatically know it was originally `skipped`.
    *   *Recommendation:* If strict invertibility is required (e.g., two-way sync with a legacy system), you must store the `original_legacy_state` alongside the new `RunState` in your database/memory. If this mapping is only for *reading* legacy data and migrating it forward, then invertibility is not actually required, and the current design is perfect.

### File 4: `packages/shared/src/gate-registry.ts` (H-04)

**Evaluation:**
*   **Domain Semantics:** Tying `required_risk_level` directly to `ToolRiskLevel` (H-11a) creates a highly cohesive domain model. 
*   **Naming:** `approval_run_start`, `quality_compile` follow strict, predictable `snake_case` conventions.
*   **Contracts:** The static registry is unambiguous.
*   **Over-engineering:** You successfully avoided over-engineering by explicitly excluding execution logic (Phase 2). This is purely a definitional layer.

**Technical Watch-out:**
*   **Compile-time Uniqueness:** The design notes mention a "compile-time uniqueness check (Map size vs array length)". TypeScript cannot easily check array *values* for uniqueness at compile time without complex recursive type mapping. 
    *   *Recommendation:* The standard way to handle this in TS is to derive a union type from the array: `type GateName = (typeof GATE_REGISTRY)[number]['name']`. If you have duplicates, TS won't complain, but a simple runtime check on module load (`new Set(gates.map(g => g.name)).size !== gates.length`) will catch it immediately during tests.

# hep-calc: LaTeX → (agent rewrite) → FeynRules model_build

> Language: English. 中文版: `references/model_build_latex.zh.md`

This document describes the **auditable, deterministic** workflow and extension points of `auto_qft.model_build`.
The goal is to mechanically extract/normalize/parse Lagrangian blocks from complex TeX sources (EFT/ChPT/etc.), while leaving the **physics mapping** to an explicitly agent-supplied `rewrite_wls`.

## Design boundary (must understand)

- The skill does not guess physics: it does not infer external sources, propagating vs non-propagating fields, conventions, or renormalization schemes.
- The skill only guarantees:
  1) TeX flattening (optional `latexpand`) + extraction (environments + display math),
  2) best-effort macro expansion (subset of preamble + overrides),
  3) syntax normalization (e.g. slash / h.c.),
  4) parsing selected blocks via Mathematica `TeXForm` into **HoldComplete** expressions,
  5) calling the agent’s `rewrite_wls` to generate a loadable `.fr` snippet.

## Relevant job fields (quick view)

See `references/job_schema.md` and `assets/job_schema.json` for the full schema; key fields:

- `auto_qft.model_build.tex_paths` (list[path]): TeX files
- `auto_qft.model_build.inline_tex` (string): a single TeX snippet (mutually exclusive with `tex_paths`)
- `auto_qft.model_build.preprocess.flatten` (bool, default true): flatten multi-file TeX with `latexpand`
- `auto_qft.model_build.preprocess.macro_overrides` (dict): override/extend macro definitions (best-effort)
- `auto_qft.model_build.selection.mode`: `lagrangian_like` (default) or `all_math_blocks`
- `auto_qft.model_build.selection.include_patterns` / `exclude_patterns`: Python regex filters applied to `normalized_tex`
- `auto_qft.model_build.parse_policy`: `best_effort` (default) or `strict`
- `auto_qft.model_build.base_model_files`: FeynRules skeleton `.fr` files (declare fields/parameters/external sources)
- `auto_qft.model_build.rewrite_wls`: agent-supplied Mathematica `.wls/.m`

## TeX extraction scope (current implementation)

Environments (line-scan for `\\begin{...}` / `\\end{...}`):
- `equation`, `align`, `multline`, `eqnarray`, `gather` (including `*` variants)

Display math (regex extraction):
- `$$ ... $$`
- `\\[ ... \\]`

Inline math `$ ... $` is currently not extracted. Workarounds:
- put relevant content into `inline_tex`, or
- use an extractor plugin / pre-processing.

## Normalization rules (parseability only; no physics meaning)

In `blocks_all.json` / `blocks_selected.json` we retain:
- `raw_tex` (original)
- `expanded_tex` (after best-effort macro expansion)
- `normalized_tex` (normalized for `TeXForm`)

Key normalizations:

- Alignment/labels cleanup:
  - remove `&`, `\\`, `\\label{...}`, `\\notag` / `\\nonumber`
- `h.c.`:
  - `\\text{h.c.}` / `\\mathrm{h.c.}` → `\\text{HC}` / `\\mathrm{HC}` (avoids `TeXForm` crashing on `.`)
- Slash:
  - `\\slashed{X}` → `\\text{Slash}(X)` → parsed by Mathematica as `Slash[X]`
  - `\\not{X}` / `\\not X` / `\\not\\!X` → `\\text{Slash}(X)`
  - `\\cancel{X}` → `\\text{Slash}(X)` (compat with some author conventions)

Note: `\\text{Slash}(...)` is only a parseable placeholder; its physics meaning must be defined by the agent in the rewrite step.

## parse_policy: best_effort vs strict

- `best_effort` (default):
  - lines failing `TeXForm` parsing are listed under `model_build/summary.json.warnings`
  - `parsedBlocks` is still passed to `rewrite_wls` (including lines with `ok=false`)
  - the agent decides whether the failure is fatal (e.g. ignore noisy lines, or raise an explicit error)
- `strict`:
  - any selected line failing `TeXForm` parsing aborts model_build (useful for strict consistency audits)

## rewrite_wls: interface and responsibilities (agent side)

`rewrite_wls` must define:

- `HepCalcModelBuildRewrite[parsedBlocks_, ctx_]`

Where:
- `parsedBlocks` is a list of elements containing:
  - `"id"`, `"file"`, `"env"`, `"labels"`
  - `"normalized_tex"` (string)
  - `"ok"` (bool)
  - `"expr_held"`: `HoldComplete[...]` (or `HoldComplete[$Failed]`)
- `ctx` is an Association containing keys like `job/out_dir/auto_dir/model_build_dir/lagrangian_symbol/parse_policy`

Recommended return (Association):
- `"lagrangian"` (required): a Mathematica expression acceptable to FeynRules
- `"lagrangian_symbol"` (optional): override the exported symbol name
- `"notes"` / `"warnings"` (optional): written into the audit summary

Templates and runnable examples:
- Template: `assets/model_build/rewrite_template.wls`
- Example (SM identity / reuse `LSM`): `assets/model_build/rewrite_sm_identity.wls` + `assets/demo_auto_qft_model_build_sm_identity.yml`

Typical agent responsibilities:
- decide which blocks/terms enter the effective Lagrangian (e.g. include all `\\mathcal{L}_2, \\mathcal{L}_4` blocks)
- handle external/background “source” symbols (e.g. `l_\\mu, r_\\mu, s, p, ...`) and declare them as parameters in the base model (non-propagating)
- provide explicit mappings for `Slash[...]`, traces, chiral building blocks, etc. (the skill will not infer these)
- explicitly state any additional rules (custom macro handling, units/sign conventions, etc.)

## Security model (required disclosure)

`rewrite_wls` is agent/user-supplied Mathematica code executed via `Get[...]` (no sandbox).
Only run it if you trust its source. In shared environments, prefer running Mathematica inside a container or under an isolated account.

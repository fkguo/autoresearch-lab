# research-writer — M9D-plan-r2 Review Packet (add `draft_sections` as opt-in)

## Goal
Decide whether it is appropriate to add an **opt-in** `draft_sections` command to `research-writer` that helps an agent draft paper sections while enforcing the anti-hallucination evidence gate.

Interpretation for this plan review:
- `VERDICT: READY` means: it is appropriate to proceed implementing `draft_sections` as a **skill feature** now, with the specified guardrails.
- `VERDICT: NOT_READY` means: do not implement as a skill feature now (keep agent-only), or the plan needs major redesign.

## Motivation (real failure mode)
We saw hallucinated additions about dataset provenance and uncertainty modeling (e.g., “taken from NNOline”, “uniform errors manually added”) without evidence anchors. This breaks trust.

A centralized opt-in skill command can reduce this risk vs agent-only ad-hoc drafting because it can:
- always inject guardrails,
- always run the deterministic linter post-generation,
- always write trace logs.

## Scope boundary (must not violate)
- MUST NOT auto-mutate `assets/style/physics_discussion_logic_playbook.md`.
- Drafting is opt-in (no surprise model calls).
- Default writes to a staging area (never overwrite `paper/main.tex`).

## Proposed deliverable (M9D)
Add:
- `scripts/bin/research_writer_draft_sections.py` (new)
- docs updates to `RUNBOOK.md` + `SKILL.md`
- smoke coverage in `scripts/dev/run_all_smoke_tests.sh`

### Core workflow (proposed)
Inputs:
- `--project-root` (research-team project)
- `--paper-dir` (generated by scaffold) OR `--out` for drafts
- context sources: `Draft_Derivation.md`, `knowledge_base/`, `artifacts/` (selected by `--tag`)

Outputs (default):
- `paper/drafts/<run_id>/<section>.tex` (staging; never overwrites main)
- `paper/drafts/<run_id>/trace.jsonl` (or `trace.json`) recording:
  - input paths + hashes
  - guardrails prompt path/hash
  - model + timeouts
  - raw model output path
  - evidence-gate linter result + flagged snippets (if any)

Safety enforcement:
- The script MUST load and apply `assets/style/research_writer_guardrails_system_prompt.txt`.
- The script MUST run `scripts/bin/check_latex_evidence_gate.py --fail` on generated `*.tex` drafts.
- If the evidence gate fails:
  - do NOT write in-place to `paper/main.tex`.
  - keep the draft in staging but mark it clearly (e.g., rename to `*.unsafe.tex` and/or write `EVIDENCE_GATE_FAILED.md`).

Modes:
- `--section <intro|formalism|results|discussion>` (single section) and `--all`.
- Optional dual-model mode (no auto-merge): write `drafts/claude_<section>.tex` and `drafts/gemini_<section>.tex`.
- Offline smoke mode: `--stub-models` writes deterministic placeholder drafts and still runs the linter.

## Acceptance criteria (for implementing M9D)
- [ ] New CLI exists: `scripts/bin/research_writer_draft_sections.py` and `--help` works.
- [ ] Default behavior never modifies `paper/main.tex`.
- [ ] Staging output path is deterministic and auditable (run dir + section files).
- [ ] Guardrails prompt is always applied (in code, not by convention).
- [ ] Evidence-gate linter is always run; failures are clearly surfaced and block any in-place writes.
- [ ] Trace log is always written; schema documented.
- [ ] Smoke tests (offline) cover:
  - help works,
  - stub draft generation,
  - a deliberately unsafe draft triggers evidence-gate failure,
  - a properly anchored draft passes,
  - idempotency of trace/log writing.

## Reviewer request
Approve or reject implementing M9D as a skill feature based on safety, auditability, and real-workflow fit.
